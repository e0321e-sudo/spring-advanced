# 🚀 [Spring 3기] CH 3 심화 Spring_코드 개선 과제
기존 코드의 버그를 수정하고, 아키텍처를 개선하며, 성능을 최적화하는 과정을 기록했습니다.

---

## 🛠 과제별 해결 과정 (Step-by-Step)

### **Lv 0. 초기 설정 및 환경 구축**
* **내용**: 프로젝트 가동을 위한 인메모리 데이터베이스(H2) 설정 및 JWT 보안 키 세팅.
* **해결 과정**:
    1. `build.gradle`에 프로젝트 필수 의존성을 구성하고, 가벼운 테스트 환경을 위해 H2 데이터베이스 연결 설정을 수행함.
    2. 서버 보안을 위해 JWT(`jwt.secret.key`)와 토큰 만료 시간 설정을 `application.yml`에 체계적으로 관리하도록 구축함.
* **성과**: 복잡한 외부 설정 없이도 즉시 개발과 테스트가 가능한 독립적인 백엔드 환경을 완성함.

### **Lv 1. Argument Resolver를 활용한 인증 자동화**
* **내용**: `@Auth` 어노테이션과 `AuthUserArgumentResolver` 구현 및 `WebConfig` 등록.
* **성과**: 컨트롤러 메서드 파라미터에서 로그인한 유저 정보(`AuthUser`)를 즉시 주입받도록 자동화함. 반복적인 토큰 파싱 코드를 제거하여 비즈니스 로직의 응집도를 높임.

### **Lv 2. 코드 개선 및 가독성 확보**
* **Early Return 적용**: `AuthService.signup`에서 이메일 중복 체크를 로직 최상단으로 이동. 불필요한 암호화(Encoding) 연산을 방지하여 서버 자원을 효율화함.
* **Refactoring**: `WeatherClient` 등에서 불필요한 `else` 문을 제거하여 코드의 깊이(Depth)를 줄이고, 읽기 좋은 평탄한 코드로 개선함.
* **Validation 통합**: 서비스 레이어의 복잡한 `if` 검증 로직을 DTO의 `@Valid` 어노테이션으로 이관하여 유지보수성을 향상함.

### **Lv 3. N+1 문제 해결 및 성능 최적화**
* **문제 정의**: Todo 목록 조회 시 연관된 User 정보를 가져오기 위해 Todo 개수만큼 추가 SELECT 쿼리가 발생하는 N+1 문제를 확인했습니다. 데이터가 늘어남에 따라 쿼리 실행 횟수가 기하급수적으로 증가하는 성능 병목 지점을 인지했습니다.
* **해결 방안**: 조회 방식의 특성에 따라 최적화 도구를 이원화하여 적용했습니다.
  1. **목록 조회 (`findAll`)**: `@EntityGraph(attributePaths = {"user"})`를 사용하여 페이징(`Pageable`) 처리를 유지하면서도 `User` 엔티티를 한 번의 조인으로 함께 조회하도록 개선했습니다.
  2. **단일 조회 (`findById`)**: `@Query` 문에 `FETCH JOIN`을 명시하여 필요한 시점에 즉시 연관 데이터를 로딩하도록 처리했습니다.
  * **안정성 확보**: `LEFT JOIN` 방식을 활용하여 연관된 `User` 정보가 없는 `Todo` 데이터도 누락 없이 안전하게 조회되도록 설계했습니다.
* **성과**: 기존 10건 조회 시 최대 11번 발생하던 쿼리를 **단 1번의 통합 JOIN 쿼리**로 최적화하여 데이터베이스 부하를 대폭 감소시켰습니다.

### **Lv 4. 테스트 코드 신뢰성 확보 및 버그 수정**
* **PasswordEncoder**: `matches()` 메서드의 인자 순서 오류를 바로잡아 인증 로직 검증을 정상화함.
* **Exception Sync**: 테스트가 기대하는 예외 타입과 서비스 예외 타입을 일치시켜 테스트 신뢰도를 확보함.
* **NPE 방어**: `ManagerService`에서 발생할 수 있는 `NullPointerException`을 조건문 우선순위 조정을 통해 안정적으로 해결함.

### **📝 회고 (Retrospective)**

이번 과제에서 가장 큰 수확은 단순히 기능을 구현하는 것을 넘어, **로그를 통해 시스템의 성능 병목을 추적하고 방어적인 대안을 도출**해 본 경험이었습니다.

1.  **동작 원리의 시각화**: 인텔리제이 콘솔 로그를 통해 `N+1` 문제가 실제로 발생하는 과정을 확인하고, 최적화 후 `Join` 쿼리 한 줄로 통합되는 과정을 직접 검증하며 JPA의 동작 원리에 대한 이해도를 높였습니다.
2.  **트러블슈팅과 대안 탐색**: 페이징 처리가 포함된 컬렉션 조인 시 발생할 수 있는 **메모리 과부하(In-memory Paging) 경고**의 위험성을 인지했습니다. 이를 방어하기 위해 현재의 `ManyToOne` 관계 조인을 넘어, 향후 대용량 데이터 환경에서 필수적인 **`@BatchSize` 전역 설정**을 통한 최적화 방안을 함께 검토했습니다.
3.  **데이터 무결성 고려**: `LEFT JOIN`과 `INNER JOIN`의 차이를 고려하여 작성자가 없는 특수한 상황에서도 데이터가 유실되지 않도록 설계하는 등, 꼼꼼한 예외 상황 방어의 중요성을 체감했습니다.
